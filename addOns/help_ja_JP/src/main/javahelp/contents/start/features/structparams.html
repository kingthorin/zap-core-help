構造パラメーター

構造パラメーターは構造修飾子の一種であり、ユーザーデータではなくアプリケーションの構造を表すパラメーターを識別します。

従来型のWebアプリケーションでは、アプリケーションの構造は通常、URLパスによって定義され、データはURLパラメーターとPOSTデータに含まれます。
以下のようなURLがあるとします：

  • https://www.example.com/app/aaa?ddd=eee
  • https://www.example.com/app/aaa?ddd=fff
  • https://www.example.com/app/aaa?ddd=ggg
これらは サイトツリー において、ツリー内の一つの「ノード」として表現されます：
  • サイト
    • https://www.example.com
      • app
        • GET:aaa(ddd)
サイトツリーは、ZAPがアプリケーション構造を理解するための非常に重要な要素です。
サイトツリーが構造を適切に表現していない場合、ZAPはアプリケーションを効果的に攻撃できません。

「シングルページ」アプリケーションでは、論理的な「ページ」を示すためにパラメーターが使用されます：
  • https://www.example.com/app/aaa?page=p1&ddd=eee
  • https://www.example.com/app/aaa?page=p2&ddd=fff
  • https://www.example.com/app/aaa?page=p3&ddd=ggg
これら3つのURLは異なる論理ページを表しますが、デフォルトではZAPはこれらを依然として一つのノードとして表現します：
  • サイト
    • https://www.example.com
      • app
        • GET:aaa(ddd,page)
これは問題です。なぜなら、このままではZAPはアプリケーションの全ての機能を攻撃しないからです。

ZAPの用語では、「page」URLパラメーターは「構造パラメーター」、つまりアプリケーション構造の一部を定義するパラメーターです。
構造的パラメーターを定義するには、まずアプリケーションを コンテキスト に追加し、次に セッションコンテキスト構造画面経由で設定します.
この設定を行うと、ページは以下のように3つのノードとして正しく表現されるようになります：
  • サイト
    • https://www.example.com
      • app
        • aaa
          • GET:p1(ddd,page)
          • GET:p2(ddd,page)
          • GET:p3(ddd,page)
