データ駆動型コンテンツ

データ駆動型コンテンツは、データを表すURLパス要素を識別する構造修飾子の一種です。

従来型のWebアプリケーションでは、アプリケーションの構造は通常、URLパスによって定義され、データはURLパラメーターとPOSTデータに含まれます。
以下のようなURLがあるとします：

  • https://www.example.com/app/aaa?ddd=eee
  • https://www.example.com/app/aaa?ddd=fff
  • https://www.example.com/app/bbb?ddd=eee
は、 サイトツリー 内で2つの「ノード」として表現されます：
  • サイト
    • https://www.example.com
      • app
        • GET:aaa(ddd)
        • GET:bbb(ddd)
サイトツリーは、ZAPがアプリケーション構造を理解するための非常に重要な要素です。
サイトツリーが構造を適切に表現していない場合、ZAPはアプリケーションを効果的に攻撃できません。

一部のアプリケーションでは、URLパス内にデータが含まれます。
例えば：
  • https://www.example.com/app/company1/aaa?ddd=eee
  • https://www.example.com/app/company2/aaa?ddd=fff
  • https://www.example.com/app/company3/aaa?ddd=ggg
これら3つのURLは、異なるデータ (company1、company2、company3) に対して同じ機能を呼び出しています。しかし、デフォルトではZAPはこれらを3つの別々のノードとして認識します：
  • サイト
    • https://www.example.com
      • app
        • company1
          • GET:aaa(ddd)
        • company2
          • GET:aaa(ddd)
        • company3
          • GET:aaa(ddd)
これは問題です。なぜなら、ZAPは実際には1つのページだけを攻撃すれば良いのに、すべてを攻撃します。
この例では、同じページを3回攻撃してもそれほど大きな問題ではありませんが、このようなページが数百、数千ある場合、アプリケーションのスキャンにかかる時間が大幅に増加します。

ZAPの用語では、「company」ノードは「データ駆動型コンテンツ」です - アプリケーション構造の一部を表すのではなく、データを含むURLパス要素です。
データ駆動型コンテンツは、アプリケーションを コンテキスト に追加し、 サイトタブ の「コンテキストとしてフラグ付け → コンテキスト名 : データ駆動型ノード」右クリックメニュー項目から設定することで定義できます。
この設定を行うと、ページは正しく1つのノードとして表現されます：
  • サイト
    • https://www.example.com
      • app
        • «company»
          • GET:aaa(ddd)
文字 « と » は、これが「特殊な」ノードであることを示すために使用され、ノード名 (この場合は「company」) は、そのノードが何を表しているかを示すために設定できます。
