認証方式

ZAPは、Webサイト/Webアプリに使用できる複数のタイプの認証 (認証方式と呼ばれます) を処理します。 各コンテキストには、認証の処理方法を定める認証方式が定義されます。 認証は、認証されたWebアプリケーションのユーザーに対応するWebセッションを作成するために使用されます。

認証方式は、ZAP内の複数の場所で使用されます。 主な用途は以下の通りです：

  • ユーザーの定義と自動ログイン
  • 認証済み/未認証状態の検出
  • 自動再認証の実行

複数の認証方式が実装されており、システムはユーザーのニーズに応じて新しい方式を簡単に追加できるようになっています。 主なものを以下に説明します。

手動認証

この方式では、ユーザーが手動で認証を実行します (例えば、ZAPをプロキシとして経由させながらブラウザで認証する)。その後、対応するHTTPセッションを選択します。 注意: 実際の認証はユーザー自身が実行するため、この方式ではWebアプリケーションがユーザーをログアウトさせた場合の自動再認証には対応していません。 ただし、認証検証戦略を定義している場合、ZAPは認証統計情報を維持します。

この認証方式を使用する場合、コンテキストのユーザーを設定するには、認証済みHTTPセッションを選択する必要があります。

フォームベース認証

この方式は、フォームの送信またはGETリクエストを「ログインURL」に実行することによって認証が行われるWebサイト/Webアプリで使用されます。認証には「ユーザー名/パスワード」のペアが使用されます。 再認証が可能です。 設定は、セッションコンテキスト認証画面を使用して行うか、右クリックメニューのフラグ設定... フォームベース認証ログインリクエストを使用して行うことができます。

この認証方式を使用する場合、コンテキストのユーザーを設定するには、フォームベース認証に使用されるユーザー名/パスワードのペアを設定する必要があります。

アプリケーションがログインページに提示されたAnti-CSRFトークンの送信を必要とする場合、ZAPは自動的に処理します。 トークン名がオプション Anti-CSRF画面で設定されていることを確認してください。

JSONベース認証

この方式は、「ログインURL」にJSONオブジェクトを送信することによって認証が行われるWebサイト/Webアプリで使用されます。認証には「ユーザー名/パスワード」のペアが使用されます。 再認証が可能です。 設定は、セッションコンテキスト認証画面を使用して行うか、右クリックメニューのフラグ設定... JSONベース認証ログインリクエストを使用して行うことができます。

この認証方式を使用する場合、コンテキストのユーザーを設定するには、認証に使用されるユーザー名/パスワードのペアを設定する必要があります。

HTTP/NTLM認証

この方式は、HTTPメッセージヘッダーを使用したHTTPまたはNTLM認証メカニズムで認証を行うWebサイト/Webアプリで使用されます。 サポートされている認証スキームは、Basic、Digest、NTLMの3つです。 認証ヘッダーは認証済みリクエストごとに送信されるため、再認証が可能です。 設定はセッションコンテキスト認証画面を使用して行うことができます。

この認証方式を使用する場合、コンテキストのユーザーを設定するには、HTTP/NTLM認証に使用されるユーザー名/パスワードのペアを設定する必要があります。

スクリプトベース認証

この方式は、認証がより複雑で、認証プロセスを処理するカスタムスクリプトが有益なWebサイト/Webアプリに役立ちます。 この方式を使用するには、まず、Webアプリケーションで必要に応じてメッセージを送信したり、他のアクションを実行したりする認証スクリプトを定義する必要があります。 このスクリプトは、特定のコンテキストで使用するために選択され、認証が実行されるたびに呼び出されます。 再認証が可能です。 設定はセッションコンテキスト ダイアログを使用して行うことができます。

この認証方式を使用する場合、コンテキストのユーザーを設定するには、スクリプトで定義されたパラメーターのセットを設定する必要があります。 詳細については、提供されている認証スクリプトの例を参照してください。

設定方法
